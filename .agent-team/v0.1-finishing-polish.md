## W-1: magna-remote field-generation glue

**Where:** `c:/git/fellwork/magna/crates/magna-remote/src/`

**Status:** The crate skeleton exists with public types (`RemoteResolver`, `RemoteError`, `WebhookRequest`, `build_client`) and 2 unit tests. The missing piece: turning a `Vec<RemoteResolver>` into actual GraphQL fields registered via `magna_build::SchemaExtension`.

**Goal:** A `RemoteExtension` struct that implements `SchemaExtension` and wires each configured `RemoteResolver` as a Query or Mutation field that POSTs to the configured webhook on resolution.

**Read first:**

- `crates/magna-build/src/extension.rs` — the `SchemaExtension` trait, `ExtensionContext` (especially `query_field`, `mutation_field`, `add_data`), `replace_with` helper. Understand the `register_types` → `extend_query` → `extend_mutation` phase contract.
- `crates/magna-remote/src/lib.rs` — the existing skeleton.
- `crates/magna-build/tests/extension_wiring.rs` — the 7 wiring tests show the expected idioms (Arc<Mutex<HashSet>> side-channel for cross-extension verification, journaled phase ordering, parametric has_mutations gating).
- The api-side `crates/fw-resolvers/src/extension.rs` — the reference Tier 1 implementation pattern for how an extension holds state and threads it into resolver closures via `Arc::clone`.

**Concrete shape:**

```rust
pub struct RemoteExtension {
    resolvers: Vec<RemoteResolver>,
    client: reqwest::Client,
}

impl RemoteExtension {
    pub fn new(resolvers: Vec<RemoteResolver>) -> Result<Self, RemoteError> {
        let client = build_client()?;
        Ok(Self { resolvers, client })
    }
}

impl SchemaExtension for RemoteExtension {
    fn name(&self) -> &str { "magna-remote" }

    fn extend_query(&self, ctx: &mut ExtensionContext<'_>) {
        for r in &self.resolvers {
            if r.field.starts_with("Query.") {
                ctx.query_field(self.build_field(r));
            }
        }
    }

    fn extend_mutation(&self, ctx: &mut ExtensionContext<'_>) {
        for r in &self.resolvers {
            if r.field.starts_with("Mutation.") {
                ctx.mutation_field(self.build_field(r));
            }
        }
    }
}
```

The `build_field(&self, r: &RemoteResolver) -> Field` method is the meat. It needs to:

1. Parse `r.field` to get the GraphQL field name (strip `Query.` / `Mutation.` prefix).
2. Parse `r.returns` into an `async_graphql::dynamic::TypeRef` (e.g. `"WeatherReport"` → `TypeRef::named("WeatherReport")`, `"[Order!]!"` → `TypeRef::named_nn_list_nn("Order")`, etc.).
3. Build a resolver closure that:
   - Captures `Arc<reqwest::Client>` and the `RemoteResolver` config by clone.
   - Reads `args` from the GraphQL `ResolverContext` and serializes to JSON.
   - Reads `Authorization` header from the request context if `forward_auth: true` (use `ctx.data_opt::<HeaderMap>()` if you've stored one — check what magna-serv puts in schema-data).
   - POSTs to `r.url` with `WebhookRequest { args, context, parent }` body and the configured timeout.
   - On 2xx: deserializes the JSON response into an `async_graphql::Value` and returns `Ok(Some(value))`.
   - On non-2xx or transport failure: returns a structured GraphQL error with `r.url` and the upstream status redacted.
   - Wraps the call in a `tracing::info_span!("magna-remote.call", field = %r.field, url = %r.url)`.

**Scope boundaries (what NOT to do):**

- **Don't** support nested input objects in `args_schema` — flat scalar args only at v0.1. Document this in the field's GraphQL description.
- **Don't** introduce dependencies on `serde_yaml`. The YAML loader is `magna-config`'s job; this crate consumes already-deserialized `RemoteResolver` structs.
- **Don't** add retry logic or circuit-breakers. v0.2 concern.
- **Don't** expose anything new on `magna-build`'s public API. If you find yourself wanting to add a method to `ExtensionContext`, that's a v0.2 design conversation — surface it and stop.
- **Don't** chase the type-ref parser into supporting every GraphQL type expression. v0.1 supports `Foo`, `Foo!`, `[Foo]`, `[Foo!]`, `[Foo!]!`. Anything else: return `RemoteError::InvalidUrl` (rename if needed) at extension-construction time, not at request time.

**Tests to add** (in `crates/magna-remote/src/lib.rs` `#[cfg(test)]` or a sibling integration test):

1. `RemoteExtension::new` constructs a non-empty extension successfully.
2. `register_types` is no-op (RemoteExtension doesn't introduce types — webhook-typed return types must be registered by another extension or pre-exist in the schema).
3. End-to-end against a `mockito` or `wiremock` test server: configure one Query field, build a schema with `RemoteExtension` + a tiny `register_types`-only extension that registers the return type, execute a GraphQL query, assert the response matches what the mock returned.
4. Auth-forwarding test: configure `forward_auth: true`, set an `Authorization` header in the request context, assert the mock saw it. Toggle to `false`, assert it didn't.
5. Timeout test: configure a 100ms timeout, mock a 500ms-delayed response, assert the resolver returns a structured error (not panics).

**Acceptance test (the "this is done" gate):**

```bash
cd c:/git/fellwork/magna
cargo test -p magna-remote --all-targets
```

Expected: all tests pass, including the end-to-end mock-server one. Then update `docs/feature-matrix.md` Tier 3 row from `[YES]` (currently optimistic) to `[YES]` with a file pointer to the new field-generation code, and add a `### Added` line to `CHANGELOG.md`'s Unreleased section.

**Risk:** The async-graphql 7 dynamic schema's resolver closures have non-obvious lifetime requirements. If the closure capture turns into a Send bounds issue, the fix is usually to clone `Arc<reqwest::Client>` outside the closure and capture the clone. The existing magna-build code (`crates/magna-build/src/resolve/query.rs`) shows the pattern.
